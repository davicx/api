## Architecture principles (read before implementing)

These decisions apply across Part 1 and future mutations (RDS, IAM, Terraform, etc.).

### Layering

```text
Atlas        = execution engine (real AWS mutations, source of truth for runs — eventually)
Navigator/API = orchestration + workflow + UI shaping (chat fields, confirmation, operation_id)
Kite         = generic renderer
```

Do not describe Atlas as something to “wire up” — it is already the execution layer. Stages **connect Navigator orchestration to Atlas**, not “make Atlas real.”

### Mutation template

`createEC2Handler` is the **canonical mutation handler**. Clone the pattern for toggle and delete so response shape, execution-mode gating, and workflow behavior stay aligned. This is the emerging **CloudPilot mutation architecture** — intentional, not over-engineered.

### Action tiers

`informational` vs `destructive` (with `executionModes`) scales better than per-action special cases. Keep using it.

`executionMode === "automatic"` gates real AWS today. Later variants (`automatic_safe`, `automatic_with_rollback`, `automatic_with_monitoring`) can extend without redesign.

### `operation_id` (add in Stage 0.5 — cheap now)

Every mutation should carry an `operation_id` (e.g. `op_xxx`) generated by Navigator when execution starts. Pass through execution context and Atlas request body.

**Not persisted yet** — no DB/table required for MVP. Introduce the **concept and field** now for: audit, retries, status polling, timeline, race prevention, websockets, rollback (later).

### Shared mutation metadata (Stage 0.5)

Standardize on execution context (in-memory only for MVP):

```js
{
  operation_id,
  execution_mode,
  initiated_by,
  requested_at
}
```

Handlers and Atlas calls should accept/pass these consistently before create/toggle/delete diverge further.

### Workflow state vs execution lifecycle

`ActionState` (in-memory) is **temporary chat workflow state** only — fine for MVP.

**Do not** tie long-running execution lifecycle to `ActionState`. Toggle can take minutes; a Node restart can drop workflow state while Atlas still runs.

Eventually: **Atlas execution records** (table or equivalent) become source of truth for run status; ActionState remains chat UX only.

Target execution record shape (future, not MVP):

```json
{
  "operation_id": "op_xxx",
  "action_type": "toggle_ec2",
  "execution_mode": "automatic",
  "status": "running",
  "started_at": "...",
  "completed_at": null,
  "rollback_available": false
}
```

### Toggle: MVP vs product

| Phase | How toggle resolves targets |
|-------|---------------------------|
| **MVP (Stages 2–4)** | Explicit `primary_instance_id` + `secondary_instance_id` in workflow (acceptable for E2E) |
| **Product (Stage 5+)** | Tag/role-based: e.g. `cloudpilot-role: primary` / `secondary` in region — “toggle production API” without raw IDs from user |

`cloudpilot-role` on create defaults is groundwork for tag-driven toggle. Do not lock the product into raw instance IDs forever.

### Rollback

**Do not build rollback in this track.** Sequence: Create → Toggle → Delete → stable mutation flow → **then** rollback.

Rollback needs operation history, snapshots, idempotency, partial failure recovery, and execution persistence — roughly 2× mutation complexity. Thinking about `operation_id` early is correct; implementing rollback is not.

### Destructive UX (later)

Before final confirmation on destructive actions, add a clear danger boundary (copy only), e.g. “This action may modify AWS infrastructure.” Not required for first E2E pass.

### Explicitly deferred

- Rollback / undo
- Execution persistence DB
- Kite mutation UI
- Per-user AWS credentials

### Implementation rules (non-negotiable)

Use this checklist for every change in Part 1 (Stages 0–4). Do not drift.

- [ ] **Do not** implement rollback or an execution DB in this track.
- [ ] **Do** add `operation_id` + shared mutation metadata in **Stage 0.5** before cloning toggle/delete handlers.
- [ ] **Clone** `createEC2Handler` for toggle and delete — same structure, gates, and response mapping. **Do not** invent new handler patterns.
- [ ] **MVP toggle** uses explicit instance IDs in workflow/API body; add comments (and light structure) in Atlas toggle core pointing toward **tag/role resolution** in Stage 5 — do not treat raw IDs as permanent product design.
- [ ] **Atlas unreachable** (`ATLAS_BASE_URL` down or fetch failure) must return a controlled error for create, toggle, and delete (e.g. `atlas_unreachable`) — no unhandled fetch crashes.
- [ ] **ActionState** is for chat workflow only (fields, mode, confirmation). **Do not** use it to track long-running execution status; execution lifecycle belongs on Atlas/execution records later.

## Part 1: EC2 Mutations (Create / Delete / Toggle)

### Goal

All three actions run end-to-end through the CloudPilot chat workflow in **`automatic`** execution mode:

```text
User confirms
  → AtlasExecution
  → createEC2Handler | deleteEC2Handler | toggleEC2Handler
  → atlasEC2Functions (fetch)
  → Atlas POST /ec2/create | /ec2/delete | /ec2/toggle
  → boto3 (AWS profile "atlas")
  → AWS account updated
```

**Reference implementation:** `create_ec2` (~70% done). Mirror for delete and toggle.

**Out of scope for this track:** Kite UI, iOS, doc-folder cleanup, rollback, execution DB, full delete safety policy (Stage 5+).

### Current reality (audit 2026-05-24)

| Operation | Atlas (Python) hits AWS? | API / CloudPilot chat wired? | Notes |
|-----------|--------------------------|------------------------------|-------|
| **Create** | Yes — `run_instances` in `manage_instances.py` | Yes — `createEC2Handler` → `POST /ec2/create` | Blocked without `image_id` or `DEFAULT_AMIS` (both regions empty today) |
| **Delete** | Yes — `terminate_instances` | **No** — not in `actionRegistry`, no handler, no `atlasEC2Functions.deleteEC2` | Works only via direct Atlas HTTP |
| **Toggle** | Yes — stop primary, start secondary + waiters | **No** — `toggleEC2Handler` calls OpenAI only in automatic mode | Works only via direct Atlas HTTP; bad IDs can 500 (unhandled boto errors) |

**Registry today:** `actionRegistry.js` has `create_ec2`, `toggle_ec2`, `scan_ec2`, `inventory_aws`, `general_chat`. No `delete_ec2`.

**Workflow fields today:**

- `create_ec2`: `name`, `region`, `instance_type` — OK
- `toggle_ec2`: `region` only — MVP will add instance IDs; long-term → tag/role resolution (see [Toggle: MVP vs product](#toggle-mvp-vs-product))
- `delete_ec2`: not defined

**Tagging:** Create merge documented in `api/doc/instructions/cloudpilot_tagging_metadata.md`. Delete safety and toggle metadata called out as later.

**Related Atlas paths:**

```text
atlas/app/api/routes/ec2_operation_routes.py
atlas/app/api/services/ec2_operation_service.py
atlas/app/core/cloud/ec2/operations/manage_instances.py   # create, delete
atlas/app/core/cloud/ec2/operations/toggle_instances.py   # toggle
api/application/atlas/functions/actions/ec2/atlasEC2Functions.js
api/application/atlas/functions/actions/ec2/createEC2/createEC2Handler.js
api/application/atlas/functions/actions/ec2/toggleEC2/toggleEC2Handler.js
```

### Prerequisites (before coding stages)

- [ ] Atlas running at `ATLAS_BASE_URL` (default `http://127.0.0.1:8000`)
- [ ] AWS profile `atlas` with EC2 permissions (`atlas/app/config/config.py` → `PROFILE = "atlas"`)
- [ ] Test region (e.g. `us-west-2`) aligned with workflow defaults
- [ ] Known AMI for create in that region (`DEFAULT_AMIS` or explicit `image_id`)
- [ ] Two real instance IDs for toggle E2E (primary stopped / secondary started scenario)

### Stage 0 — Atlas hardening (shared foundation)

**Purpose:** Consistent errors and create unblock before more Node wiring.

| Task | Detail |

| 0.3 Response contract | Confirm `/ec2/create`, `/ec2/delete`, `/ec2/toggle` all use same envelope handlers expect (`success`, `message`, `data`, `errors`) |
| 0.4 Direct Atlas smoke tests | curl/Postman valid + invalid bodies — no Node |

**Exit criteria:** All three Atlas endpoints succeed or fail with structured JSON.

### Stage 0.5 — Shared mutation metadata (Navigator)

**Purpose:** One shared contract before handlers diverge. No persistence.

| Task | Detail |
|------|--------|
| 0.5.1 Generate `operation_id` | Navigator/API when execution is requested (e.g. `op_` + uuid/slug) |
| 0.5.2 Execution context | Attach `operation_id`, `execution_mode`, `initiated_by`, `requested_at` to `AtlasExecution` payload / handler `context` |
| 0.5.3 Pass to Atlas | Include `operation_id` (and optional metadata) in create/toggle/delete request bodies — Atlas can log now, persist later |
| 0.5.4 Handler template | Document required fields on `createEC2Handler` so toggle/delete copy the same metadata pass-through |

**Exit criteria:** All three mutation handlers receive and forward the same metadata shape; Atlas logs or echoes `operation_id` in responses.

## Part 2: Navigator UI Response Contract

### Goal

Keep this mental model:

```text
Atlas = raw truth
Navigator/API = UI shaping layer
Kite = generic renderer
```

Lightweight extensibility: new Atlas shapes get small adapters; Kite renders generic primitives (`stats`, `tables`, `alerts`, …).

### Current status

The API/Navigator contract and first adapters are **in place**.

**Helper:**

```text
application/atlas/functions/navigatorResponseFunctions.js
```

**Adapters (implemented):**

```text
application/atlas/functions/actions/aws/inventoryAWS/atlasAWSInventoryNavigatorAdapter.js
application/atlas/functions/actions/ec2/scanEC2/atlasEC2ScanNavigatorAdapter.js
```

**API behavior:** Additive — existing formatted Atlas response remains; `navigatorResponse` added for generic UI.

**Wired handlers:**


**Not wired:** `create_ec2`, `toggle_ec2` (no mutate adapters yet; optional in EC2 Stage 5).

### Future work

#### Future — Persist Navigator data (React Query cache)

Kite stores latest Navigator data in React context. Next step: React Query cache so dashboard data survives navigation without refetch.

- Cache key: `["navigator-data", groupID, conversationID]`
- Rules: React Query for API/cache; no Redux; no localStorage unless refresh persistence needed
- Options later: `staleTime`, `cacheTime`, `refetchOnWindowFocus: false`

**Owner:** Kite/client module.

#### Future — More Navigator adapters

When Atlas adds a new **read** response type, add a small adapter. Examples: cost, security, compliance.

Do not change Kite if existing primitives suffice.

**Mutating actions** (`create_ec2`, `delete_ec2`, `toggle_ec2`): adapters optional — see [EC2 Stage 5](#stage-5--hardening-post-mvp).

### Architecture rules (Navigator)

1. **Atlas** — raw truth only; no tables/cards/UI metadata.
2. **Navigator/API** — owns `meta`, `stats`, `tables`, `cards`, `alerts`, `actions`, raw opt-in.
3. **Kite** — generic primitives only; no service-specific field names.
4. **Mappings** — prefer explicit `row_id`, `key` fields over `Object.keys` spread.

### Design goal

```text
new Atlas data shape → small API adapter → same Navigator contract → same Kite renderer
```
